## 1) 🧱 Основа: что такое SHA‑256 и почему ему доверять

**SHA‑256** — криптографический «отпечаток» файла: строка из **64** шестнадцатеричных символов (256 бит).  
Один и тот же файл на любой машине даёт один и тот же SHA‑256; изменение **хотя бы одного байта** полностью меняет хэш.

**Что это даёт при аудите:**
- ✅ **Целостность:** совпадение хэша означает, что каждый бит проверяемого файла идентичен исходнику.  
- 🔁 **Повторяемость:** результат не зависит от платформы/ПО.  
- 🧾 **Манифесты:** удобно хранить строки вида `ХЭШ␣␣относительный/путь/к/файлу` и проверять **весь набор** разом.

**Криптографические свойства (без формул):**
- 🌩️ **Эффект лавины:** мельчайшая правка входа полностью меняет выход.  
- 🧩 **Стойкость к коллизиям:** практически невозможно найти **два разных файла с одним и тем же SHA‑256**.  
- 🎯 **Стойкость к предобразу:** практически невозможно **подобрать файл под заранее заданный SHA‑256**.

**Почему «практически невозможно»?**  
Пространство значений — около **2^256** (≈ 1,16×10^77). Полный перебор нереалистичен физически. Даже при **квантовом ускорении (алгоритм Гровера)** запас прочности для задач проверки целостности остаётся колоссальным.

> ⚠️ Ограничение: SHA‑256 подтверждает **целостность**, но сам по себе не отвечает на **кто** опубликовал артефакт и **когда** он существовал. Эти аспекты закрывают **GPG‑подпись** (подлинность) и **OTS‑якорь** (датировка).

## 2) 🧾 Рекомендуемый формат манифеста SHA‑256

Одна строка на файл:
```
<64-символьный-хэш><ДВА ПРОБЕЛА><относительный/путь/к/файлу>
```

Пример:
```
d2a5…9b0f  dist/app-1.0.0.tar.gz
f1c3…7a21  docs/README.md
```

Почему так:
- ␠␠ **Два пробела** — стандарт для `sha256sum`/`shasum -c`.  
- 📦 **Относительные пути** — удобнее переносить репозиторий целиком.

---

## 3) 🗝️ Что добавляет GPG‑подпись (подлинность: «кто»)

**GPG‑подпись** — криптографический «автограф» файла (обычно подписывается **манифест**).  
Подпись создаётся **приватным ключом** издателя; проверяется по соответствующему **публичному ключу**.

**Успешная проверка подписи означает:**
- файл **не менялся** с момента подписания;  
- подписал его **владелец соответствующего приватного ключа** (тот, чей публичный ключ используется при проверке).

**Критично:** нужна уверенность, что **публичный ключ действительно авторский**.  
Сверяется **fingerprint** (отпечаток ключа) с тем, что объявлен заранее по независимым каналам (официальный сайт/репозиторий/релиз‑ноты). Совпал fingerprint → ключ считается авторским.

### 🧭 Коротко: если публичный ключ скачан из репозитория
1) **Показать fingerprint без импорта**:  
   ```bash
   gpg -n --import-options show-only --import PUBLISHING_KEY.asc
   # или
   gpg --show-keys --with-fingerprint PUBLISHING_KEY.asc
   ```
2) **Сверить fingerprint по независимому каналу.** Любое расхождение → остановить проверку.  
3) **Импортировать и проверить подпись**:  
   ```bash
   gpg --import PUBLISHING_KEY.asc
   gpg --verify manifest.sha256.asc manifest.sha256
   ```
   Зачёт только если есть `Good signature …` **и** fingerprint точно совпадает.

(Опционально при ротации ключа: `gpg --check-sigs <NEW_FPR>` — новый ключ должен быть подписан старым.)

---

## 4) ⛓️ Что добавляет OTS‑якорь (время: «когда»)

**OpenTimestamps (OTS)** привязывает хэш (обычно — **хэш манифеста**) к **времени в публичном блокчейне Bitcoin**.  
Файл `manifest.sha256.ots` — компактное криптодоказательство того, что данный хэш существовал **не позже** конкретного блока цепи. Доказательство не раскрывает содержимое артефактов (в цепь уходит только хэш).

**Роли слоёв в сумме:**

| Слой | Что доказывает | От чего защищает |
|---|---|---|
| **SHA‑256** | Бит‑в‑бит **целостность** файлов | Случайная порча/тихая модификация |
| **GPG‑подпись** | **Подлинность/авторство** манифеста | Подмена файлов и хэшей третьей стороной |
| **OTS‑якорь** | **Дата/время** существования хэша (не позже времени блока) | Фальсификация таймлайна публикации |

Комбинация трёх уровней делает подделку практически невозможной: потребовалось бы одновременно подобрать коллизию SHA‑256, скомпрометировать приватный GPG‑ключ и переписать историю блокчейна.

---

## 5) 📦 Что обычно передаётся аудиторам

- **Артефакты**: файлы релиза (`…tar.gz`, `…zip` и т. п.).  
- **Манифест хэшей**: `manifest.sha256`.  
- **GPG‑подпись манифеста**: `manifest.sha256.asc` (detached, ASCII‑armored).  
- **OTS‑якорь для манифеста**: `manifest.sha256.ots`.  
- **Публичный GPG‑ключ**: `PUBLISHING_KEY.asc` **и/или** его **fingerprint**, опубликованный заранее по независимым каналам.

## 6) 🧪 Проверка SHA‑256 (команды для всех ОС)

**macOS**:
```bash
# один файл
shasum -a 256 путь/к/файлу
# весь манифест
shasum -a 256 -c manifest.sha256
```
(Альтернатива: `gsha256sum -c manifest.sha256` из coreutils.)

**Linux**:
```bash
sha256sum -c manifest.sha256
# или
sha256sum путь/к/файлу
```

**Windows — PowerShell**:
```powershell
# один файл
Get-FileHash -Algorithm SHA256 -Path "C:\path	oile"

# по манифесту (двойной пробел обязателен)
Get-Content .\manifest.sha256 | ForEach-Object {
  if ($_ -match '^\s*([0-9a-fA-F]{64})\s\s(.+)$') {
    $hash = $matches[1].ToLower()
    $path = $matches[2]
    $got  = (Get-FileHash -Algorithm SHA256 -Path $path).Hash.ToLower()
    if ($got -eq $hash) { "OK     $path" } else { "FAILED $path" }
  }
}
```

**Windows — Git Bash / WSL**: как на Linux (`sha256sum -c manifest.sha256`).

Ожидается `OK` по всем строкам; `FAILED` означает байтовые отличия.

---

## 7) 🗝️ Проверка GPG‑подписи (подлинность)

**macOS / Linux / Windows (GnuPG/Gpg4win):**
```bash
# импорт публичного ключа
gpg --import PUBLISHING_KEY.asc

# сверка отпечатка (fingerprint) с объявленным заранее
gpg --fingerprint

# проверка подписи манифеста
gpg --verify manifest.sha256.asc manifest.sha256
```
Засчитывается, если `Good signature …` и fingerprint ключа **точно совпадает** с объявленным.

---

## 8) ⏱️ Проверка OTS‑якоря (время) — включая случай с несколькими блоками

**Автоматически** (если установлен ots‑клиент):
```bash
ots verify manifest.sha256.ots
```

**Если `ots verify` не сработал** (нет сети/клиента, доказательство «не дозрело»):
```bash
ots info manifest.sha256.ots
```
Замечания по `ots info`:
- Вывод может содержать **несколько якорей/аттестаций**, то есть **несколько блоков** (разные календари могли якорить в разное время).  
- **Проверяются все указанные блоки/txid** через независимый обозреватель (например, **mempool.space**): найти `block height/hash` или `txid`, убедиться, что блок существует в mainnet и имеет корректную временную метку.  
- Для консервативной датировки берётся **самый ранний по времени валидный блок** из списка — это минимальная верхняя граница «существовало не позже».

При наличии сети можно «дозавершить» доказательство и повторить верификацию:
```bash
ots upgrade manifest.sha256.ots
ots verify manifest.sha256.ots
```

---

## 9) 🧰 Почему хэши архивов иногда расходятся и как это учитывается

SHA‑256 зависит от **каждого бита**. Архивы включают **метаданные** (порядок, владельцы/группы, метки времени, xattrs, параметры сжатия).  
Чтобы одинаковые архивы давали одинаковые хэши на разных машинах, артефакты упаковываются **детерминированно** (фиксируются `mtime`, владельцы/группы, порядок, отключаются лишние метаданные, gzip — без таймстампа). В таком режиме расхождения хэшей не ожидаются.

---

## 10) ✅ Сводный порядок проверки (TL;DR)

1) **SHA‑256**:  
   `shasum -a 256 -c manifest.sha256` (macOS) / `sha256sum -c manifest.sha256` (Linux/WSL/Git Bash) / скрипт PowerShell (Windows).  
   Ожидается `OK` по всем строкам.

2) **GPG‑подпись**:  
   `gpg -n --import-options show-only --import PUBLISHING_KEY.asc` → сверка fingerprint по независимому каналу → `gpg --import PUBLISHING_KEY.asc` → `gpg --verify manifest.sha256.asc manifest.sha256`.  
   Ожидается `Good signature …` и точный матч fingerprint.

3) **OTS‑якорь**:  
   `ots verify manifest.sha256.ots`; при затруднениях `ots info manifest.sha256.ots` и **проверка всех указанных блоков** в обозревателе (например, **mempool.space**).  
   За «раннюю» метку времени принимается **самый ранний по дате валидный блок**.

Если все три шага проходят, получено криптографически сильное подтверждение, что:
- файлы идентичны исходным по байтам (целостность, SHA‑256);  
- манифест подписан авторским ключом (подлинность, GPG);  
- манифест существовал не позже зафиксированного времени (датировка, OTS), причём при множественных якорях учитывается **самый ранний**.

# Быстрая проверка SHA-256 без «перебазирования» путей

Этот раздел для тех, кто **не хочет возиться** с переводом абсолютных путей в относительные и прочими тонкостями.  
Просто посчитайте хэш у себя и сравните **64-символьную** строку с опубликованной в репозитории.  
(Регистр `A–F`/`a–f` не важен — можно приводить к нижнему.)

---

## ✅ Один файл → один хэш

**macOS**
```bash
# хэш + имя файла
shasum -a 256 /path/to/file
# только 64-символьный хэш
shasum -a 256 /path/to/file | awk '{print $1}'
```

**Linux**
```bash
sha256sum /path/to/file
sha256sum /path/to/file | awk '{print $1}'
```

**Windows (PowerShell)**
```powershell
# хэш + путь
Get-FileHash -Algorithm SHA256 -Path "C:\path	oile"
# только 64-символьный хэш
(Get-FileHash -Algorithm SHA256 -Path "C:\path	oile").Hash
```

Далее откройте в репозитории соответствующий опубликованный хэш (или файл с хэшем) и **визуально сравните** вашу 64-символьную строку с опубликованной.

---

## 🔁 Сверка «файл A vs файл B» (должны быть идентичны)

**macOS / Linux**
```bash
# Вариант 1: пошагово
A=$(shasum -a 256 /path/to/A | awk '{print tolower($1)}')
B=$(shasum -a 256 /path/to/B | awk '{print tolower($1)}')
[ "$A" = "$B" ] && echo "MATCH" || echo "DIFFERENT"

# Вариант 2: одной строкой
[ "$(shasum -a 256 /path/to/A | awk '{print tolower($1)}')" = "$(shasum -a 256 /path/to/B | awk '{print tolower($1)}')" ] && echo "MATCH" || echo "DIFFERENT"

# Примечание: на Linux можно использовать sha256sum вместо shasum:
# A=$(sha256sum /path/to/A | awk '{print tolower($1)}')
# B=$(sha256sum /path/to/B | awk '{print tolower($1)}')
```

**Windows (PowerShell)**
```powershell
$A = (Get-FileHash -Algorithm SHA256 -Path "C:\path	o\A").Hash.ToLower()
$B = (Get-FileHash -Algorithm SHA256 -Path "C:\path	o\B").Hash.ToLower()
if ($A -eq $B) { "MATCH" } else { "DIFFERENT" }
```

## 📜 Проверка по манифесту (вручную, «как есть»)

1. Откройте `manifest.sha256` (в редакторе или прямо в репозитории).  
   Формат строки:
   ```
   <64-символьный-хэш><ДВА ПРОБЕЛА><относительный/путь/к/файлу>
   ```
2. Найдите нужный файл и **скопируйте его 64-символьный хэш** из манифеста.
3. У себя локально посчитайте хэш этого файла (команды выше) и **сравните 64 символа**.

### (Необязательно) Проверка одной строки из манифеста командой

**macOS / Linux**
```bash
EXPECTED="paste_64_hex_here"
ACTUAL=$(shasum -a 256 /path/to/file | awk '{print $1}')
[ "${ACTUAL,,}" = "${EXPECTED,,}" ] && echo "OK" || echo "FAILED"
```

**Windows (PowerShell)**
```powershell
$Expected = "PASTE_64_HEX_HERE"
$Actual   = (Get-FileHash -Algorithm SHA256 -Path "C:\path	oile").Hash
if ($Actual.ToLower() -eq $Expected.ToLower()) {"OK"} else {"FAILED"}
```

---

## ℹ️ Полезные подсказки

- **Ровно 64 символа.** Убедитесь, что при копировании нет лишних пробелов/переводов строки.  
- **Регистр неважен.** Сравнивайте в одном регистре (нижнем), как в примерах.  
- **Несовпадение = другой файл.** Даже 1 байт разницы даёт совершенно иной хэш.  
- **Никаких «ребейзингов».** Для этого способа **не нужно** переводить пути в относительные — просто возьмите нужный файл, посчитайте хэш и сравните с опубликованным значением.

# Проверка SHA‑256‑манифестов с абсолютными путями 

Иногда опубликованный `artifacts.sha256` создан с **абсолютными путями** (например, `/Users/name/.../file.svg`).  
**Хэши при этом корректны** — путь в строке служит лишь «ярлыком» для `sha256sum -c`, чтобы найти файл.  
Ниже — как проверить подписи и хэши на **любой машине**.

### TL;DR
1) Проверьте **подпись GPG** исходного `artifacts.sha256`.  
2) Создайте **локальную копию** манифеста с **относительными именами** и выполните `sha256sum -c` по этой копии.

### 1) Проверка подписи (исходный манифест)
```bash
gpg --verify artifacts.sha256.asc artifacts.sha256
# (опционально) OpenTimestamps
ots verify artifacts.sha256   # или: ots verify artifacts.sha256.ots
```

### 2) «Перебазирование» путей локально и проверка хэшей

**Linux / macOS (bash)**
```bash
sed -E 's#^([0-9a-f]{64})  .*/([^/]+)$#  #' artifacts.sha256 > artifacts.local.sha256
sha256sum -c artifacts.local.sha256
```
> Команда `sed` сохраняет 64‑символьный hex‑хэш и заменяет путь на одно имя файла.

**Windows (PowerShell)**
```powershell
Get-Content artifacts.sha256 | % {
  if ($_ -match '^([0-9a-f]{64})\s\s(.+)$') {
    "$($matches[1])  $(Split-Path $matches[2] -Leaf)"
  }
} | Set-Content artifacts.local.sha256

# В Git Bash / WSL:
# sha256sum -c artifacts.local.sha256
```

### Примечания и FAQ
- **Почему «не совпадает» автоматически?**  
  `sha256sum -c` пытается открыть файл **по пути из манифеста**. Если путь абсолютный и его нет на вашей машине — будет `No such file or directory`. **Сами хэши при этом верны**.
- **Влияет ли путь на хэш?**  
  Нет. SHA‑256 считается **только от содержимого** файла.
- **Можно проверить один файл вручную?**  
  Да:
  ```bash
  sha256sum somefile.svg
  # Сравните 64‑символьный хэш с первой колонкой в artifacts.sha256
  ```
- **Практика GPG/OTS**  
  Подпись **GPG** проверяйте на исходном `artifacts.sha256`. Для удобства держите рядом обе версии:
  - `artifacts.sha256` (исходный, подписанный; может содержать абсолютные пути)  
  - `artifacts.local.sha256` (ваша локальная копия для проверок `-c`)

### (Опционально) macOS без GNU‑утилит
```bash
# Создать простой отсортированный манифест с относительными именами
ls -1 | grep -v '^artifacts\.sha256$' | LC_ALL=C sort | xargs shasum -a 256 > artifacts.local.sha256

# Проверка отдельного файла:
shasum -a 256 somefile.svg
```
